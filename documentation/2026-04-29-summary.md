# Third Space: 2026-04-29 Summary

This is a simple write-up of what was done today, made for someone who knows embedded systems but not web development.

## What this app does

Third Space is a tiny shared note page.
- You open it in a browser.
- You type in a box.
- The browser sends your text to a local program running on your computer.
- That program keeps the latest text in `data/space.json`.
- With a temporary public link, your friends can also open the same page and edit it.

If the server is not available, the browser can still keep your text on your own device using browser storage. That means the page still works a little bit even when the shared server is down.

## What changed today

### 1. Made the app shareable

- We made it so the app can run on your PC and be given a public web address.
- The public address comes from a tool called `localtunnel`.
- That tunnel sends visitors to your local server while it is running.

### 2. Made the app more reliable when the shared server is gone

- If the browser cannot talk to the local server, it will use the browser's own storage.
- That storage is called `localStorage`.
- It keeps your text locally so it is not lost.

### 3. Added live update support

- When one person saves new text, the browser can get a message and update automatically.
- This is better than having to refresh the page every time.
- The server now supports a simple live channel called `Server-Sent Events`.

### 4. Included a local Node runtime in the repo

- Normally `node` is installed system-wide.
- Today, we put a copy of Node into `./.node/` inside this project.
- That means you can run the app even if your computer does not already have Node installed.
- We also made sure `.node/` is ignored by git so it does not get committed.

### 5. Added a GitHub Pages option

- The app can also be published as a website from the `public/` folder.
- GitHub Pages can serve only files like pages and code; it cannot run the server program.
- That means the static version can show the page, but it cannot save shared text to `data/space.json`.

## What the main files do

### `server.js`

This is the backend program.
- It is the code that runs on your machine.
- It sends the page files like `index.html` to the browser.
- It answers the browser when it asks for or saves text.
- It keeps the shared text in `data/space.json`.
- It now also supports a live update channel at `/api/space/events`.

### `public/index.html`

This is the page structure.
- It tells the browser what to show: the text box, the status area, and the labels.

### `public/styles.css`

This is the visual design.
- It controls colors, spacing, layout, and how the page looks on a phone.

### `public/app.js`

This is the browser code.
- It runs after the page loads.
- It asks the server for the current shared text.
- It sends your typed text back to the server.
- It also uses browser storage when the server is unavailable.
- It listens for live update messages so the page can refresh automatically.

### `data/space.json`

This is the storage file.
- It stores the text that everyone is sharing.
- It also stores the last time the text was saved.

### `.github/workflows/deploy-pages.yml`

This is a helper for GitHub.
- It tells GitHub to publish the `public/` folder as a website when the code is pushed to `main`.
- The published site cannot run the server.

## Breaking down the code itself

This section explains what the code is doing in a little more detail.

### `server.js` code concepts

- It starts a small web server on your computer.
- When the browser asks for a file like `index.html`, it reads that file from `public/` and sends it back.
- When the browser asks for `/api/space`, it reads `data/space.json` and sends the text back.
- When the browser sends text to `PUT /api/space`, it writes the new text into `data/space.json`.
- When the browser connects to `/api/space/events`, the server keeps that connection open and sends messages when the text changes.

### `public/app.js` code concepts

- It starts by loading the current shared text from `/api/space`.
- When you type, it waits a short time and then sends the current textarea contents to the server.
- It keeps track of whether the browser is currently editing or saving.
- It also checks the server every few seconds in case someone else changed the text.
- It keeps local copies in browser storage when the server cannot be reached.

### `public/index.html` code concepts

- It defines the page elements: a heading, the textarea, the status label, and the character counter.
- It includes the browser code file `app.js` so the page becomes interactive.

### `public/styles.css` code concepts

- It sets colors, fonts, and spacing.
- It makes the page look good on phones and desktop screens.

### `data/space.json` code concepts

- It is a plain text file.
- It stores only two pieces of information: the current text and the time it was last updated.

### How the code pieces work together

- The browser code and the server code talk through simple web requests.
- The browser asks for data, and the server sends data back.
- The browser sends data, and the server saves it.
- The shared file `data/space.json` is the only place the data is permanently stored.

## How the app works, step by step

### When you open the page

1. Your browser asks the local program for `index.html`.
2. The browser loads the page structure, styling, and browser code.
3. The browser asks the local program for the latest shared text.
4. The browser shows that text in the textarea.

### When you type something

1. The browser waits a short moment.
2. It sends the new text to the local program.
3. The local program writes the text to `data/space.json`.
4. The browser shows `Live` when the save is complete.

### When someone else changes the text

1. The browser is checking the server regularly, or
2. It receives a live update message from the server.
3. If you are not typing, the browser updates the textarea automatically.

### When the server is not reachable

1. The browser uses its own local storage.
2. The app still works on your device.
3. It does not stay shared with friends until the server returns.

## Important points for you

- The app has two halves: the browser side and the server side.
- The browser side runs in the browser.
- The server side runs on your computer.
- The browser can show the page by itself, but saving and sharing needs the server.
- The public tunnel only works while your computer and the server are running.

## Beginner-friendly glossary

### Browser
The program you use to look at web pages, such as Chrome or Safari.

### Server
A program that answers questions from the browser and stores data.
In this project, `server.js` is the server.

### Node
A tool that lets JavaScript run on your computer instead of inside a browser.
It is what runs `server.js`.

### HTML
The code that defines the structure of a web page.
`index.html` is the file that tells the browser what elements to show.

### CSS
The code that defines how the page looks.
`styles.css` controls colors, spacing, and layout.

### JavaScript
The code that makes the page do things.
`app.js` is JavaScript that runs in the browser.
`server.js` is JavaScript that runs on the computer.

### JSON
A plain-text format for data.
`data/space.json` stores the shared text in a simple format.

### Local storage
A place inside the browser where the page can keep data on your device.
It is used when the server cannot be reached.

### GitHub Pages
A service that can host only page files.
It cannot run the server program.

### Tunnel
A temporary public web address that sends visitors to your local computer.
It lets friends reach your local app while it is running.

### URL
A web address, like `https://huge-lemons-brush.loca.lt`.

### Static site
A website made only of files that the browser loads directly.
It does not run server code.

### API
A way for the browser and the server to talk.
In this app, the browser uses `/api/space` to get and save text.
